## Цель работы

Цель нашей работы — подготовить раздел детской энциклопедии **«Я и цифровой мир»**, который объясняет подросткам и детям младшего возраста, как устроена цифровая среда, какие в ней есть возможности и риски, и как взаимодействовать с ней безопасно, осознанно и с пользой.

При выполнении работы мы использовали:

- **явное представление знаний** на основе **Wikidata**;
- **SPARQL-запросы** для извлечения фрагментов графа знаний;
- **автоматическую генерацию текстов** с помощью **LLM API**;
- **Python-скрипты** для обработки данных, генерации markdown-страниц и простановки перекрёстных ссылок.

## Типы связей в нашей онтологии

При построении модели мы использовали несколько типов связей.

### 1. Иерархические связи
Примеры:
- смартфон **является видом** устройства;
- ноутбук **является видом** устройства;
- киберспорт **является видом** игровой деятельности;
- микротранзакция **является видом** доната.

### 2. Причинно-следственные связи
Примеры:
- уведомления **отвлекают** внимание;
- скроллинг **увеличивает** экранное время;
- экранное время **влияет на** сон;
- алгоритмы рекомендаций **формируют** информационный пузырь;
- кибербуллинг **вредит** психологическому состоянию.

### 3. Функциональные связи
Примеры:
- пароль **защищает** персональные данные;
- двухфакторная аутентификация **усиливает** безопасность;
- операционная система **управляет** устройством;
- приложение **работает на** устройстве.

### 4. Поведенческие и социальные связи
Примеры:
- игровое сообщество **объединяет** игроков;
- виртуальная личность **влияет на** самопрезентацию;
- FOMO **усиливается через** социальные сети;
- онлайн-общение **дополняет** офлайн-общение.

Таким образом, наша концептуализация включает не только иерархию понятий, но и горизонтальные связи разных типов, как этого требует задание.

## Источники знаний и извлечение данных

### Основной структурированный источник
В качестве основного внешнего источника структурированных знаний использовалась **Wikidata**.

### Почему Wikidata
Мы выбрали Wikidata, потому что она:

- предоставляет открытый граф знаний;
- поддерживает SPARQL-запросы;
- позволяет находить сущности, метки, классы и связи;
- даёт возможность автоматически извлекать данные для дальнейшего использования в генерации текстов.

### Что именно извлекалось
С помощью SPARQL-запросов мы собирали:

- сущности, связанные с ключевыми понятиями нашей темы;
- русские и английские метки;
- связи между сущностями;
- локальные подграфы по стартовым терминам;
- примеры смежных понятий, полезных для расширения текстов и внутренней навигации.

## SPARQL и автоматизация работы с Wikidata

Для работы с Wikidata в репозитории подготовлены:

- тематические файлы `sparql_queries.py` в подпапках каждой подтемы;
- общий реестр запросов `scripts/query_registry.json`;
- скрипт `scripts/run_all_wikidata.py`, который автоматически:
  - читает реестр запросов,
  - отправляет запросы к endpoint Wikidata,
  - сохраняет сырые и нормализованные результаты в `data/`.

### Примеры задач, которые решались через SPARQL
- найти сущности по меткам ключевых терминов;
- собрать ближайшие связанные понятия;
- расширить тему через `instance of` / `subclass of`;
- получить локальный граф отношений вокруг выбранных понятий.

### Результаты выгрузки
Итоги выгрузки сохраняются в директориях вида:

- `WORK/ya_i_cifrovoy_mir/<topic>/data/wikidata_export.json`
- `WORK/ya_i_cifrovoy_mir/<topic>/data/wikidata/*.raw.json`
- `WORK/ya_i_cifrovoy_mir/<topic>/data/wikidata/*.normalized.json`

Эти данные затем использовались как контекст для генерации финальных текстов статей.

## Использование LLM API для генерации статей

Одно из ключевых требований задания — генерация markdown-страниц с помощью **генеративного ИИ**. В нашей работе это требование выполнено с помощью скрипта:

- `scripts/generate_web_with_llm.py`

### Общая логика генерации
Скрипт:

1. берёт описание статьи и её метаданные из `concepts.json`;
2. подбирает связанные данные из выгрузок Wikidata;
3. формирует промпт для языковой модели;
4. отправляет запрос к **LLM API**;
5. получает итоговый текст статьи;
6. сохраняет markdown-страницу в `WEB/.../concepts/`.

### Стиль промпта
При генерации текста использовался принцип: **объясни для десятилетнего ребёнка**

Это означало, что итоговый текст должен быть:
- понятным;
- доброжелательным;
- без лишней технической сложности;
- с простыми примерами;
- с короткими и ясными формулировками.

### Что записывается дополнительно
Для прозрачности и воспроизводимости процесса скрипт может сохранять:

- промпты,
- сырые ответы модели,
- промежуточные артефакты генерации.

Они размещаются в:
- `WORK/ya_i_cifrovoy_mir/<topic>/data/llm_generation/`

### Использованная модель
**LLM API провайдер:** `Groq`
**Модель:** `llama-3.1-8b-instant`
**Способ доступа:** `OpenAI-compatible API`

## Перекрёстные ссылки

В соответствии с заданием в проекте реализована автоматическая простановка перекрёстных ссылок между понятиями.

Для этого используются:

- `scripts/insert_crosslinks.py`

### Назначение
Скрипт проходит по markdown-страницам раздела и вставляет ссылки на другие понятия, если они упоминаются в тексте.

### Зачем это нужно
Перекрёстные ссылки делают энциклопедию более связной и удобной для чтения:

- читатель может переходить от одного термина к другому;
- раздел превращается в связанную сеть знаний;
- структура энциклопедии становится ближе к реальной wiki-модели.

### Ограничения
Мы учитывали, что автоматическая простановка ссылок может быть неидеальной, особенно в случаях:
- изменения слов по падежам;
- неоднозначных слов;
- уже существующих ссылок;
- технических фрагментов текста.

Поэтому кросс-ссылки автоматизированы, но при необходимости могут быть дополнительно проверены вручную.
